# ADR-003: SVA Bind Strategy for `mac_unit`

- Date: 2026-04-24
- Status: ACCEPTED (Phase 3, REQ-U-003)
- Deciders: uarch-designer, bfm-dev (representing P5 verification needs)

## Context

P5 formal verification will need SVA properties covering latency (REQ-P-002),
clear semantics (REQ-F-004 / REQ-F-004a), wrap (REQ-F-006), sticky overflow
(REQ-F-006), registered output (REQ-F-008), and reset quiescence (REQ-F-007).
P3 must decide whether assertions live in the module, in a bound module, or
in a separate `.sva` file. OPEN-2-003 defers this to P3.

## Options Considered

### Option A: In-module assertions
- **Pros**: Simplest authoring; assertions co-located with logic.
- **Cons**: Mixes verification code with synthesizable RTL; complicates code
  review; non-ideal for project style guidelines.
- **Impact**: RTL file becomes harder to read and maintain.

### Option B: Sibling `.sv` file bound via `bind`
```
bind mac_unit mac_unit_sva u_sva (.*);
```
- **Pros**: Project-preferred per OPEN-2-003 context. Compatible with both
  simulation (verilator, cocotb-driven) and SymbiYosys formal flows. Keeps
  RTL clean. SymbiYosys natively understands `bind`.
- **Cons**: One extra file; binding incantation in testbench.
- **Impact**: Standard pattern, well understood by tool chain.

### Option C: Testbench-only assertions
- **Pros**: Same testbench file authors both stimulus and properties.
- **Cons**: NOT compatible with SymbiYosys formal (which only sees the
  module + bound modules, not the TB). Fails REQ-U-003.AC-2.
- **Impact**: Eliminates formal flow entirely.

## Decision

**Option B — Sibling `mac_unit_sva.sv` bound via `bind`.**

Rationale:
- Matches project preference per OPEN-2-003 context.
- Compatible with both simulator (verilator) and formal (SymbiYosys) tooling.
- Keeps synthesizable `mac_unit.sv` clean.

## Consequences

**Positive:**
- Phase 5 formal verification can prove the listed properties using SymbiYosys
  BMC + induction.
- Phase 5 simulation regression also exercises the same properties (cocotb +
  verilator simulator).

**Negative:**
- One extra file (`mac_unit_sva.sv`) to maintain.
- Testbench top must contain the `bind` line.

**Trade-offs:**
- None significant — Option B dominates A and C on tooling compatibility.
