# ADR-002: Stage-1 Product Register Update Style (Gated)

- Date: 2026-04-24
- Status: ACCEPTED (Phase 3, REQ-U-002)
- Deciders: uarch-designer, timing-advisor

## Context

REQ-A-003 fixes the register set (5 flops) but does not prescribe the update
style for the Stage-1 product register `s1_product[15:0]`. Two functionally
equivalent options exist (and a third based on an ICG cell). OPEN-2-002
defers the final commit to P3.

## Options Considered

### Option A: Gated update
```
s1_product <= i_valid ? prod_c : s1_product;
s1_valid   <= i_valid;
```
- **Pros**: Avoids `s1_product` toggling on invalid cycles → lower dynamic
  power. Synthesizes to a flop with a 2:1 mux on the D pin (zero area cost
  vs. always-update).
- **Cons**: Introduces a feedback path through the mux that must be visible
  to lint/CDC tools.
- **Impact**: Power saving on workloads with sparse `i_valid`.

### Option B: Always-update
```
s1_product <= prod_c;
s1_valid   <= i_valid;
```
- **Pros**: Simplest synthesis; no D-pin mux.
- **Cons**: Higher switching activity on invalid cycles (s1_product toggles
  on every cycle even when consumer ignores it).
- **Impact**: Slightly higher dynamic power.

### Option C: Integrated clock gating (ICG) on s1_product flops
- **Pros**: Highest dynamic power savings; the flop's clock pin gates entirely.
- **Cons**: Requires instantiating `common/icg.sv` — violates REQ-A-009 (no
  submodule hierarchy). Adds ICG-timing verification burden.
- **Impact**: Would require re-opening REQ-A-009 (architectural change).

## Decision

**Option A — Gated update.**

Rationale:
- Saves dynamic power on invalid cycles at zero area cost.
- Avoids the ICG verification burden of Option C.
- Compatible with REQ-A-009 (no submodule).
- The mux pattern is the project-standard register-with-enable idiom.

## Consequences

**Positive:**
- `s1_product` does not toggle when the producer is sparse, reducing
  switching power proportionally to (1 - duty cycle of `i_valid`).
- Zero area / timing impact (synthesizes to identical cell).

**Negative:**
- Slightly more lint warnings if a tool flags D-pin muxes as candidates for
  ICG promotion (suppress with project-standard waiver).

**Trade-offs:**
- Power saving is only meaningful for workloads with low `i_valid` duty.
  At 100% duty, Option A and Option B are identical.
