# ADR-005: Reset Synchronization Contract (External)

- Date: 2026-04-24
- Status: ACCEPTED (Phase 3, REQ-U-005)
- Deciders: uarch-designer, rtl-architect

## Context

REQ-F-007 mandates `rst_n` is async-assert / sync-deassert and explicitly
states that "rst_n deassertion is expected to be synchronized externally".
The `mac_unit` itself does NOT contain a synchronizer. P3 must document this
assumption explicitly and decide whether to enforce it via internal
synchronization, an external contract, or a wrapper module. OPEN-2-005 defers
to P3.

## Options Considered

### Option A: Document "external synchronizer required"; do NOT instantiate one in mac_unit
- **Pros**: Matches REQ-F-007 acceptance criteria literally. Respects REQ-A-009
  (no submodule hierarchy). Zero area / timing impact on `mac_unit`. Standard
  practice for re-usable IP cores (the integrator already has a system-level
  reset distribution scheme).
- **Cons**: Integrator must be aware of the contract. CDC reviewers will see
  a free-running rst_n input that is "unsynchronized to clk" unless they
  understand the contract.
- **Impact**: One additional line in the integration documentation; no RTL
  change.

### Option B: Add a 2-flop rst_n synchronizer from common/ inside mac_unit
- **Pros**: Self-contained; integrator does not need to think about it.
- **Cons**: VIOLATES REQ-A-009 (no submodule hierarchy). Expands the implicit
  port contract in a way that may surprise downstream users. Adds 2 flops
  that are unnecessary if the integrator already supplies a synchronized reset.
- **Impact**: Would require re-opening REQ-A-009.

### Option C: Provide both `mac_unit` (no sync) and a `mac_unit_top` wrapper that contains the sync
- **Pros**: Caller can pick the version that fits.
- **Cons**: Escalates a hierarchy decision that was explicitly closed by
  REQ-A-009. Doubles the verification surface.
- **Impact**: Architectural-level change; not authorized at P3.

## Decision

**Option A — Document external synchronizer requirement; do not instantiate one inside `mac_unit`.**

Rationale:
- Matches REQ-F-007 acceptance criteria verbatim.
- Respects REQ-A-009 (no submodule hierarchy).
- Standard practice for IP cores at this size and reuse profile.

## Consequences

**Positive:**
- `mac_unit.sv` remains a single flat file with no instantiations.
- No area / timing penalty.
- Integrator retains full control over reset distribution architecture.

**Negative:**
- Integrator must drive `rst_n` through a 2-flop synchronizer (or equivalent)
  before connecting to `mac_unit.rst_n`.
- CDC tools may flag `rst_n` as a 1-source crossing into clk domain unless
  the project CDC waiver file documents it.

**Trade-offs:**
- Self-contained convenience (Option B) traded for spec compliance and
  architectural cleanliness.
